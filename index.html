시스템 설계

"기계에서 오작동을 막는 인터록 원리를,
소프트웨어 아키텍처에 이식합니다."

설계 철학

  • 메카트로닉스 전공에서 배운 인터록(Interlock) 설계 — 특정 조건이 충족되지 않으면 다음 단계로 넘어가지 못하게 막는 물리적 안전 원리를 데이터 처리 로직에 적용합니다.
  • 기능을 완성하는 것 못지않게, 예외 상황을 얼마나 꼼꼼히 다루는지를 중요하게 생각합니다.
  • 기계의 유기적인 메커니즘을 조망하던 습관을 소프트웨어에 이식하여, 전체 시스템이 유기적으로 맞물려 돌아가게 설계합니다.

실전 적용

  • 캠핑장 예약 플랫폼에서 운영 DB와 격리된 유령 테이블 구조를 독자적으로 설계
  • 모든 데이터가 수집된 후에만 트랜잭션을 실행하는 원자적 이관 로직 구축
  • 수십 차례 강제 종료 시나리오를 시뮬레이션하며 데이터 정합성을 꼼꼼히 검증
🔒

데이터 정합성

"저장 시점과 예외 상황까지 고려할 때,
비로소 완성되는 기능이라 생각합니다."

핵심 철학

  • 예약 시스템은 금전 거래와 직결되는 만큼, 데이터 오차가 고객 신뢰와 비즈니스에 미치는 영향이 크다고 생각합니다.
  • 사용자가 예약 도중 브라우저를 닫거나 네트워크가 끊기는 상황에서도 데이터 누락을 최소화하는 설계를 고민했습니다.
  • AI 도구가 제안한 표준 로직의 한계를 직접 발견하고, 로직의 밑바닥부터 직접 설계하여 해결합니다.

실전 성과

  • 실제 운영 DB와 완전히 격리된 임시 저장 공간에서 검증 후 이관하는 2단계 트랜잭션 구축
  • 삭제 시 데이터 유실 방지를 위한 논리적 삭제(Soft Delete) 적용
  • FK/PK 관계에서 발생하는 고아 데이터 문제를 사전에 차단하는 방어적 DB 설계

논리적 사고

"코드 한 줄에 항상 분명한 이유가 담겨 있습니다.
이유가 분명할 때, 무섭게 몰입하는 사람입니다."

논리 근육 만들기

  • 새로운 개념을 배울 때 예제를 그대로 따라 하기보다 동작 원리를 먼저 이해하려는 편입니다.
  • 막히는 문제는 우회하기보다 원인을 확인할 때까지 붙잡고 해결하는 편입니다.
  • 코드를 작성하기 전에 문제를 구조화하고 근거를 정리하는 과정을 먼저 거칩니다.

주변의 평가

  • "논리적인 사람" — 행동과 코드에 항상 분명한 근거가 있다는 평가
  • "편협되지 않은 생각" — 서로의 이야기를 들어보고 중심을 잡는 소통 스타일
  • 문제를 구조화하고, 비즈니스의 복잡한 예외를 명쾌한 로직으로 풀어내는 해결사
